Kubernetes 浅析：从分布式到云原生

简介

2014 年 Google 开源了 Kubernetes，十年后的今天它已经是云原生生态的事实标准。但 K8s 真正的价值不在于它能拉起容器——Docker Compose 也能——而在于它将 Google 内部十余年的大规模集群管理经验，凝练为了一套声明式、自愈、可扩展的分布式系统抽象。

这篇文章不打算写成”K8s 入门教程”，而是沿着两条线索展开：主线是”从分布式到云原生”的宏观演进——为什么容器会取代虚拟机、Borg 的设计教训如何塑造了 K8s 的架构决策、声明式 API 与控制循环为什么是更好的抽象；辅线是”从 Pod 到集群调度”的底层机制——Pod 为什么是最小调度单位而不是容器、Service 网络如何实现负载均衡、Controller Manager 和 Scheduler 如何协作维持集群的期望状态。

两条线索交织在一起，试图回答一个核心问题：为什么 Kubernetes 被认为是”分布式操作系统”？

一、容器化的前夜：为什么我们需要容器

1.1 虚拟机的困境

虚拟机解决了”一台物理机跑多个应用”的问题，但它以整个 OS 为粒度做隔离——每个 VM 携带独立的内核、init 系统、系统库，资源开销巨大。一个典型 Linux VM 启动需要 30-60 秒，占用 GB 级内存，而其中 90% 的组件与业务逻辑无关。当集群规模达到数千节点、服务实例数以万计时，VM 的资源浪费和启动延迟变得不可接受。

更深层的问题是，VM 镜像的可复现性很差：一个应用的部署依赖 OS 版本、系统库版本、配置文件路径的组合，这些隐式依赖在镜像构建时没有被锁定，导致”在我机器上能跑”的经典困境。PaaS 时代（Cloud Foundry、Heroku）试图通过 buildpack 解决这个问题，但 buildpack 本质上是平台特定的黑盒，换一个平台就需要重新适配，开发者失去了对运行环境的控制权。

1.2 Docker 的崛起与容器标准化

Docker 在 2013 年的爆发并非偶然——它抓住了两个关键创新。第一是镜像分层与 Union FS：通过 overlay/aufs 将文件系统叠加，使得镜像构建的每一步都是增量层，不同镜像可以共享相同的底层（比如都基于 ubuntu:20.04 只需要一份基础层），极大减少了存储和传输开销。第二是Dockerfile 作为构建 DSL：将”环境搭建”从运维人员的口头约定变成可版本控制、可审计的代码。

2015 年的 OCI（Open Container Initiative）将容器运行时和镜像格式标准化，Docker 不再是唯一的容器引擎，runc、containerd、CRI-O 等实现相继出现。标准化带来的真正影响是解耦：容器编排系统不再绑定特定运行时，Kubernetes 通过 CRI（Container Runtime Interface）可以对接任何符合 OCI 的运行时——这意味着 K8s 关注的是”编排”，容器本身只是被编排的工作负载单元。

二、从 Borg 到 Kubernetes：Google 的内部遗产

2.1 Borg：Google 的秘密武器

在 Docker 诞生之前近十年，Google 已经在内部大规模使用容器化技术。Borg 是 Google 的集群管理系统，2003 年前后投入生产，管理着数万台机器上的数十万个作业。Borg 引入的几个核心概念直接塑造了后来的 K8s：Cell（一组机器的管理单元，对应 K8s 的集群）、Job（一组相同任务的实例，对应 Deployment/ReplicaSet）、Alloc（一组预留资源，对应 Pod 的资源预留概念）。

Borg 的一个关键洞察是装箱（bin-packing）：调度器不是简单地为每个任务分配一整台机器，而是将多个任务压缩到同一台机器上运行，最大化集群资源利用率。Google 在 2015 年的 Borg 论文中披露，通过 Borg 的装箱调度，Google 的机器利用率达到了 60-70%，远高于当时行业 10-20% 的平均水平。这个数字对 Google 意味着每年节省数十亿美元的硬件成本。

2.2 Omega 的教训：为什么中心化调度不够

Borg 的成功也暴露了一个架构问题：中心化的 BorgMaster 负责全部调度决策，面对 Google 不断膨胀的集群规模，单体调度器成为瓶颈和单点故障。Omega 是 Google 的第二代集群管理系统（2013 年论文），它尝试了共享状态 + 乐观并发的调度模式：多个调度器并行从共享状态中读取集群快照，独立做出调度决策，通过乐观并发控制解决冲突。

Omega 的实验揭示了一个深刻的取舍：完全去中心化的调度虽然可扩展性更好，但调度质量（全局最优 vs 局部最优）显著下降。最终 Google 既没有选择纯中心化（Borg）也没有选择纯去中心化（Omega），而是在两者之间找到了一个平衡点——Kubernetes 的 Scheduler 就是这一权衡的产物：调度器是单一逻辑组件但无状态，可水平扩展，每个调度决策基于最新的 etcd 快照，调度结果通过 API Server 写入 etcd，形成最终一致性。

2.3 Kubernetes 的诞生与设计决策

2014 年 K8s 开源时，Google 做出了一个反直觉的决定：它没有直接将 Borg 的代码开源，而是从零用 Go 重写。原因在于 Borg 是用 C++ 编写的，与 Google 内部的 Chubby（锁服务）、Bigtable 等基础设施深度绑定，无法剥离。更重要的是，Borg 是为”Google 内部场景”优化的——所有应用由 Google 自己编写，Job 的生命周期被高度约束——而 K8s 需要面向一个开放的多租户、多语言、多框架的世界。

K8s 从 Borg 继承了什么，又放弃了什么？继承的：Pod 抽象（对应 Borg 的 Alloc）、声明式任务描述（对应 Borg 的 Job config）、标签选择器（对应 Borg 的 constraint language）、滚动更新机制。放弃的：Borg 的优先级抢占（在 Borg 中高优先级任务可以直接驱逐低优先级任务，K8s 用更温和的 PriorityClass + 优雅终止来替代）、Borg 的硬性资源预留（K8s 用 request / limit 做软约束）。这些取舍反映了从”内部封闭系统”到”开放平台”的思维转变——K8s 需要保障多租户间的安全边界，而非最大化单一组织的资源效率。

三、Kubernetes 的设计哲学

3.1 声明式 API vs 命令式 API

K8s 最根本的设计选择是选择了**声明式（Declarative）**而非命令式（Imperative）的交互模型。命令式 API 的理念是”告诉我怎么做”——kubectl run、kubectl scale、kubectl delete 每一步操作都由用户显式触发。声明式 API 的理念是”告诉我你想要什么状态”——你提交一份 YAML 描述期望的 Pod 数量、镜像版本、资源配置，系统自己计算达到该状态需要执行哪些操作。

声明式的优势在分布式环境中尤为明显。如果你通过 kubectl scale --replicas=5 扩容，然后某个 Pod 意外被节点故障杀掉，系统不会自动恢复——因为你的命令已经执行完毕。但如果你提交的是 replicas: 5 的声明，即使节点故障导致 Pod 数量掉到 3，Controller 会发现”当前状态 ≠ 期望状态”，自动拉起新的 Pod。声明式将”一次性操作”变成了”持续保障”，这是 K8s 自愈能力的根基。

3.2 控制循环：Reconciliation Loop

声明式 API 的引擎是控制循环（Reconciliation Loop），也称为”调谐循环”。它的逻辑极其简单，只有三步：

  1. 读取资源的当前状态（from API Server / etcd）
  2. 与期望状态（spec 字段）做 diff
  3. 执行操作缩小差异，然后回到步骤 1

这个看似简单的循环有精妙之处：它是幂等的。无论 reconcile 步骤执行一次还是十次，最终状态都会收敛到期望状态。如果某次操作失败（节点宕机、网络超时），下一次循环会重新检测到差异并重试，不需要复杂的分布式事务回滚机制。它是最终一致性的工程实现——中间状态可能短暂偏离，但系统会在有限时间内自动修复。

Reconciliation Loop 的另一个好处是解耦了”检测”和”修复”：触发 reconcile 的事件可以来自 API Server 的资源变更通知（watch）、定时轮询、甚至外部系统的回调，Controller 本身不需要知道事件来源，只需要根据当前快照执行 reconcile 逻辑。

3.3 API Server 作为集群中心

在 K8s 架构中，所有组件的通信都经过 API Server——kubelet 向 API Server 上报节点状态、Scheduler 从 API Server 获取待调度的 Pod 并回写调度结果、Controller Manager 通过 API Server 读取资源状态并执行 reconcile。API Server 是集群唯一的”数据总线”，所有组件之间不存在直接的点对点通信。

这种中心化总线的设计有明确的工程收益。第一是审计与安全：所有操作都经过 API Server，认证（AuthN）、授权（AuthZ）、准入控制（Admission Control）可以在一个统一网关执行，不存在旁路。第二是解耦：组件的扩缩容和替换不需要通知其他组件，只要它们遵循 API Server 的 watch 协议。第三是可扩展性：CRD（Custom Resource Definition）+ Operator 模式让用户可以在不修改 K8s 核心代码的情况下扩展 API，CRD 的资源生命周期由自定义 Controller 管理，与原生资源（Pod、Service）无差别。

四、云原生的核心理念

4.1 微服务与不可变基础设施

云原生的定义在 CNCF 的官方文档中被概括为”使组织能够在现代动态环境中构建和运行可扩展应用的技术栈”，实际上它背后有两条核心原则。

第一条是微服务：将单体应用拆分为独立部署、独立伸缩的小服务。这不是什么新概念（SOA 时代就有人提），但容器为微服务提供了”标准化打包”——每个微服务被打包为一个容器镜像，携带精确的环境依赖。K8s 的 Service 和 Ingress 提供了服务发现与流量管理，让微服务之间的调用不需要硬编码 IP 和端口。

第二条是不可变基础设施：服务器（或容器）被部署后不应被修改。如果需要更新配置或打补丁，不是 SSH 进去改文件，而是构建新的镜像、销毁旧容器、启动新容器。这条原则消灭了”配置漂移”——没有任何一台服务器会与它的初始化状态产生不被追踪的偏离。K8s 的 Deployment 滚动更新天然支持不可变基础设施模型：每次发布都是新 ReplicaSet 的创建和旧 ReplicaSet 的缩容。

4.2 声明式管理与 GitOps

GitOps 将声明式 API 从”集群内运行”延伸到了”开发流程中”。核心理念是：Git 仓库是期望状态的唯一真实来源（Single Source of Truth）。所有的 K8s YAML 文件存储在 Git 中，CI/CD 工具（如 ArgoCD、Flux）持续监控 Git 仓库的变更，自动将集群状态同步到仓库中的声明。

GitOps 的好处是多层次的：变更历史天然具备版本控制和审计追踪（谁、什么时候、改了什么）；回滚就是一次 git revert 加自动同步；多集群管理可以通过同一个 Git 仓库的不同分支或目录结构来组织；安全合规团队可以在 PR 阶段做 YAML 的静态检查，而非事后审计。

但 GitOps 也不是银弹。它假设所有状态都能被声明式描述，而现实中某些操作（如数据库迁移、金丝雀发布的分阶段流量切换）仍然需要命令式控制。实践中 GitOps 通常与渐进式交付工具（如 Argo Rollouts、Flagger）配合，在声明式的基础上嵌入流程编排。

五、Pod：被低估的抽象

5.1 为什么不是直接调度容器

初学者最常问的一个问题是：为什么 K8s 不直接调度容器，而是引入 Pod 这一层抽象？ 答案在于”一组紧密耦合的进程需要共享资源”。假设你有一个 Web 服务，它需要一个本地的日志采集 sidecar 和请求代理 envoy——这三个容器必须部署在同一台机器上，共享网络命名空间（通过 localhost 通信）和存储卷（日志文件），并且作为一个整体被调度。

如果把容器直接暴露为调度单位，每调度一个容器都需要独立分配 IP、独立做资源预留、独立管理生命周期。而你真正想要的是一个”原子调度单元”：三个容器一起被调度到同一节点、一起分配 IP 和端口空间、一起被销毁。Pod 就是这个原子单元——Pod 内的所有容器共享 Linux Namespace（网络、IPC、可选的 PID），并且始终被调度到同一台节点上。

5.2 Pod 内的共享命名空间

Pod 的实现基于 Linux namespace 的共享。每个 Pod 启动时会先创建一个 Infra 容器（也叫 pause 容器），这个容器只做一件事——持有网络命名空间和 IPC 命名空间。业务容器通过 --net=container:<pause_id> 加入 Infra 容器的网络命名空间，从而共享同一个 IP。这就是为什么 Pod 内所有容器可以通过 localhost 互相访问。

存储卷的共享则通过 Volume 挂载点实现。Pod spec 中定义的 Volume 可以被 Pod 内的任意容器挂载到各自容器的文件系统中，多个容器可以读/写同一个目录。这种设计使得 sidecar 模式成为可能：比如一个容器写日志到 /var/log/app/，sidecar 容器读取该目录并将日志转发到集中式日志平台。

5.3 Sidecar 模式与 Pod 设计模式

Pod 的共享模型催生了几种被广泛采用的”Pod 设计模式”：

  • Sidecar 模式：辅助容器与主容器共享 Pod，做一些主容器不关心的横切关注点——日志采集（Fluentd sidecar）、服务网格代理（Envoy/Istio sidecar）、配置热加载（config-reloader sidecar）。Sidecar 对主容器透明，可以独立升级。
  • Ambassador 模式：代理容器作为主容器访问外部服务的”大使”。例如应用容器始终连接 localhost:6379，ambassador 容器负责将这个连接代理到实际的后端 Redis 集群（可能涉及分片、认证、TLS）。
  • Adapter 模式：统一输出格式。多个应用容器可能输出不同格式的监控指标，adapter 容器负责将它们转换为统一的格式（如 Prometheus metrics）暴露出去。

六、网络模型与 Service

6.1 Kubernetes 网络模型的基本要求

K8s 对网络提出了三个硬性要求，所有 CNI 插件必须满足：所有 Pod 可以在不使用 NAT 的情况下与其他所有 Pod 通信；节点上的代理（kubelet）可以与所有 Pod 通信；Pod 看到的自己的 IP 与其他 Pod 看到的一致。这意味着不存在 Pod 级 NAT——Pod 的 IP 是集群内全局可路由的。

这个模型的优点在于简化应用开发：开发者不需要关心 Pod 跑在哪个节点上，Pod IP 在整个集群内是平坦可达的。代价是网络插件需要负责跨节点的 Pod 通信，通常通过 overlay（Flannel VXLAN、Calico IPIP）或 underlay（Calico BGP、Cilium eBPF）来实现跨主机的 Pod 互联。选择哪种方案取决于性能需求（overlay 有封装开销）和网络策略复杂度。

6.2 kube-proxy 与 Service 类型

Pod IP 是短暂的——Pod 被销毁重建后 IP 会变。Service 解决了”如何稳定访问一组 Pod”的问题：Service 拥有一个固定的虚拟 IP（ClusterIP），kube-proxy 在每个节点上维护 iptables/IPVS 规则，将对 ClusterIP 的流量负载均衡到后端的 Pod IP 上。

Service 有四种类型，各自解决不同场景：

  • ClusterIP：集群内部可达，不对外暴露。这是默认类型，适用于微服务之间的内部调用。
  • NodePort：在每个节点上开放一个指定端口（30000-32767），外部流量通过 <NodeIP>:<NodePort> 进入集群。简单但不适合生产——端口范围有限，依赖节点 IP 稳定性。
  • LoadBalancer：通过云厂商的 LB（如 AWS ELB、GCP Cloud LB）对外暴露。云厂商 Controller 会自动创建 LB 并配置后端为 NodePort。
  • ExternalName：将 Service 映射为一个外部 DNS 名称，不经过代理。适合将集群外部服务（如托管数据库）以 Service 的形式暴露给内部应用。

6.3 CNI 插件生态

CNI（Container Network Interface）是一个轻量级规范，只定义了 ADD/DEL 两个操作——为容器分配网络接口（ADD）和回收（DEL）。这种极简设计使得网络插件可以独立于容器运行时发展。

主流 CNI 插件代表了两种设计哲学：

  • Flannel：最简单的 overlay 方案，使用 VXLAN 或 host-gw 做跨节点通信。性能一般但配置极简，适合学习和中小规模集群。
  • Calico：使用 BGP 协议在节点间交换 Pod CIDR 路由，不需要 overlay 封装（使用 IPIP 时可选项），性能优于 overlay 方案。支持 NetworkPolicy（网络策略），可以做 Pod 粒度的防火墙规则。
  • Cilium：基于 eBPF 在内核层处理网络和负载均衡，绕过了 kube-proxy 的 iptables 规则链。在 Service 数量上千时，iptables 的 O(n) 规则匹配会成为瓶颈，而 Cilium 的 eBPF map 查找是 O(1)。

七、控制器与控制循环

7.1 Controller Manager 的架构

Controller Manager 不是单一进程在执行所有控制循环——它是一个进程内运行着多个独立 Controller 的容器。每个 Controller（Deployment Controller、ReplicaSet Controller、Node Controller、Job Controller……）各自 watch 相关资源的事件，独立执行 reconcile 逻辑。

这种”一进程多 Controller”的设计有几个好处：所有 Controller 共享同一个 API Server 连接池（减少连接数）；Controller 之间的依赖可以通过事件传播自动解决（Deployment Controller 创建 ReplicaSet → ReplicaSet Controller 检测到新 RS → ReplicaSet Controller 创建 Pod）；如果某个 Controller panic，只影响到该 Controller 的 reconcile，不会拖垮其他 Controller（每个 Controller 有独立的 recover）。

7.2 Deployment 与 ReplicaSet

Deployment 是上层抽象，ReplicaSet 是下层实现，两者共同构成了最常用的无状态工作负载管理方案。Deployment 的 spec 声明了期望的 Pod 模板和副本数，Deployment Controller 创建对应的 ReplicaSet，ReplicaSet Controller 再根据 ReplicaSet 的副本数创建 Pod。这个层次关系看起来多了一层，但它解决了滚动更新的核心问题。

滚动更新时，Deployment Controller 创建新的 ReplicaSet（带新 Pod 模板），逐步增加新 RS 的副本数，同时减少旧 RS 的副本数。如果在更新过程中检测到问题（新 Pod crash、健康检查失败），可以中止滚动并回滚——本质上是把新 RS 缩容到零，旧 RS 扩容回去。ReplicaSet 保留了历史版本，因此回滚是瞬间的（不需要重新拉镜像、调度 Pod），只需调整两个 RS 的副本数。

7.3 StatefulSet 与有状态应用

StatefulSet 解决了 Deployment 无法处理的问题：有状态应用需要稳定的网络标识和持久存储。一个 3 副本的 Deployment 中的 Pod 名字是随机的（web-5d8f7c9b6-xk2j），Pod 被替换后名字和 IP 都变了。但一个 3 副本的 StatefulSet 中的 Pod 名字是固定序号的（db-0、db-1、db-2），即使 Pod 被重新调度到其他节点，名字、DNS 记录、关联的 PVC 都不变。

StatefulSet 的另一个关键设计是有序部署和缩容。db-0 启动并通过健康检查后，db-1 才开始启动；缩容时从最高序号开始（db-2 先停止）。这个有序性对于基于主从复制的数据库（如 MySQL、PostgreSQL）至关重要——db-0 通常是主库，必须先于从库启动。

八、调度器与集群编排

8.1 Scheduler 的两阶段调度

K8s Scheduler 为每个待调度的 Pod 选择一个最优节点，分为两个阶段：

  • 过滤（Filtering / Predicates）：剔除”绝对不可能”的节点。比如节点资源不足（CPU/内存不够 Pod 的 request）、节点有污点（taint）且 Pod 无对应容忍（toleration）、节点选择器（nodeSelector）不匹配、端口冲突。经过过滤后得到一个候选节点集合。
  • 打分（Scoring / Priorities）：对候选节点打分排序。打分规则包括：节点资源使用率（尽量选择空闲节点以实现负载均衡）、Pod 亲和性/反亲和性（尽量让相关 Pod 集中或分散）、镜像本地性（节点已有镜像则拉取更快）。最终选择得分最高的节点。

Scheduler 本身是无状态的，调度结果通过 API Server 写入 Pod 的 spec.nodeName 字段。如果 Scheduler 宕机，未调度的 Pod 保持在 pending 状态，不会丢失。这种设计允许 Scheduler 做水平扩展（虽然通常只跑一个实例，但社区有多 Scheduler 方案），也允许替换为自定义调度器。

8.2 etcd：集群的”大脑”

etcd 是 K8s 集群中唯一有状态的数据存储——所有资源对象（Pod、Service、Deployment、ConfigMap、Node……）都以 Protobuf 格式序列化后存储在 etcd 中。API Server 是唯一直接读写 etcd 的组件，其他组件通过 API Server 间接操作。

etcd 的一致性保证是 K8s 自愈能力的基础：当 Controller 通过 API Server watch 到资源变更时，这个变更是已经被 Raft 共识确认的（不会丢失或回滚），Controller 可以安全地基于它做出调度决策。但这也意味着 etcd 的性能和可靠性直接决定集群的稳定上限：所有写操作都经过 Raft 共识（Leader → majority Follower 确认），在跨可用区的集群中写入延迟至少是一个 RTT。

etcd 的容量管理是一个常见的生产问题。K8s 默认不限制 etcd 中存储的 Event 数量，长时间运行的集群可能积累数百万条 Event，导致 etcd 的 DB 文件膨胀到数十 GB。解决方案包括：设置 Event 的 TTL（--event-ttl）、使用独立的 etcd 集群存储 Event（与核心资源隔离）、定期清理。

8.3 kubelet：节点的”管家”

kubelet 是每个节点上的 agent，职责可以概括为”按 API Server 的指示管理本节点上的 Pod”。它通过 API Server watch 调度到本节点的 Pod（spec.nodeName == 本节点名），然后按照 Pod spec 的容器列表依次启动容器。

kubelet 启动 Pod 的流程：拉取镜像 → 创建 Infra 容器（建立网络命名空间）→ 调用 CNI 插件为 Pod 分配 IP → 创建业务容器 → 启动健康检查探针（liveness / readiness / startup probe）。每个步骤都可能失败，kubelet 通过退避重试（exponential backoff）和状态上报来保证最终一致性。

健康检查探针是 kubelet 保障应用可用性的关键机制。Liveness probe 检测应用是否存活（失败 → kill 容器重建），Readiness probe 检测应用是否准备好接受流量（失败 → 从 Service 后端移除），Startup probe 用于启动较慢的应用（在 startup 成功前，liveness 和 readiness 被禁用，避免启动期间被误杀）。三者配合使用可以在不侵入应用代码的情况下，实现比传统进程级别监控更精准的可用性管理。
