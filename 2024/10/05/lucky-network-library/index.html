Lucky：一个 C++ 高并发网络库的设计与实现

概述

Lucky 是一个基于 C++11 开发的 Linux 高性能网络库，采用主从 Reactor 模型实现高并发 IO 处理。主 Reactor 负责接受客户端连接，从 Reactor 负责处理具体 IO 事件，通过多线程并发充分发挥多核 CPU 的性能。

代码托管在 GitHub，项目结构清晰，核心组件包括事件循环、epoll 封装、TCP 连接管理、双缓冲异步日志，以及基于此库构建的即时通讯服务器。

本文从架构设计到每个核心类的实现细节，完整解析 Lucky 的内部机制。

一、整体架构

网络库采用主从（Master-Slave）Reactor 模型，架构如下图所示。多个客户端通过主 Reactor 向服务器发起连接请求，主 Reactor 监听 listen fd，通过 Acceptor 接受连接后分发给各个 Sub Reactor。每个 Sub Reactor 独立运行在各自线程中，负责已建立连接的 IO 读写。

该架构的核心设计理念是职责分离——连接管理（accept）与 IO 处理（read/write）由不同的模块分别承担。

主 Reactor 充当调度员的角色。它持续监听监听套接字上的可读事件（即新连接到达），通过 Acceptor 完成 accept 操作后，根据当前各 Sub Reactor 的连接数量或负载情况，将新连接分配给负载最轻的那个 Sub Reactor。两个关键收益：

  • 主 Reactor 不可能成为瓶颈。它只做 accept 和一轮指针分发，不参与任何业务 IO。即便每秒数万次 accept，主线程的 CPU 开销仍然极低。
  • 每个连接绑定到固定 Sub Reactor。该连接上后续的所有读写事件都在同一个线程内按顺序处理，不需要加锁。这是实现无锁并发的基础。

此外，Sub Reactor 的数量可以在启动时指定，也可在运行中调整。连接数增长时增加 Sub Reactor，连接数回落时减少，资源利用更灵活。

关键设计要点：

  • 主 Reactor 仅处理 accept 与分发，不参与具体业务 IO
  • 每个连接绑定到特定 Sub Reactor，同一连接上的所有事件在同一线程内处理，无需加锁
  • Sub Reactor 数量可按需配置，系统可横向扩展
  • 主 Reactor 与 Sub Reactor 通过无锁队列（pending functors）通信

Filed 的生命周期管理遵守 RAII 原则：构造时自动将 fd 和回调绑定，析构时从 Epoller 中注销。这避免了忘记调用 removeFiled 导致的内存泄漏或悬空指针。

2.4 Reactor 的完整运行流程

一次完整的事件处理周期：

  1. 服务器启动，创建 EventLoop 实例
  2. 主 EventLoop 调用 loop() 进入事件循环
  3. Epoller 调用 epoll_wait() 等待事件就绪
  4. 当监听 socket 上有新连接时，epoll_wait 返回，将对应的 Filed 加入 activeFileds 列表
  5. EventLoop 遍历 activeFileds，逐个调用 Filed::handleEvent()
  6. handleEvent 根据事件类型调用对应的回调（可读、可写、关闭、错误）
  7. 回调执行完毕，循环回到第 3 步
  8. 若指定超时且无事件，epoll_wait 超时返回，EventLoop 执行 pending functors，然后继续下一轮

四、双缓冲异步日志系统

日志在高并发网络库中常被低估。直接 fprintf 到文件时，每次写操作都可能引发系统调用和磁盘 IO。在每秒处理数万条消息的服务器上，这会迅速成为全局瓶颈。

Lucky 的日志系统通过前后端分离 + 双缓冲 + 多生产者单消费者模型解决此问题。

4.1 前后端分离

  • 前端：业务线程是日志的生产者。产生日志信息后写入内存缓冲区，不操作磁盘。
  • 后端：唯一一个后台线程（Log 线程），负责从缓冲区读取数据并写入日志文件。

生产者和消费者被解耦：生产者从不等待磁盘 IO。

4.2 双缓冲机制

日志库维护多个 4MB 大小的缓冲区。核心逻辑：

每次缓冲区满时，前后端交换缓冲区指针，耗时仅在微秒级。交换时需要加锁，但锁仅保护指针的交换操作，不保护实际的缓冲区读写或文件写入。

反过来说，不加双缓冲会怎样？ 每次日志都要加锁写文件，高并发下锁争用极其激烈。用双缓冲后，99% 的情况下日志写入只是往内存中 memcpy，只有缓冲区满时才短暂加锁交换。

5.3 消息传递机制

消息传递分两种情况处理：

点对点消息：当用户 A 给用户 B 发送消息时，服务器先在本机查找用户 B 是否在本地。如果在同一台服务器上，直接通过 Lucky TCP 连接转发。如果用户 B 在其他服务器上，则通过 Redis 发布订阅跨服务器传递。

群组消息：每创建一个群组，后端服务器订阅一个对应的 Redis 频道（频道名 group:{groupId}）。群成员发送消息时，服务器将消息发布到该频道，所有订阅该频道的服务器实时收到并推送给各自节点上的群成员。

5.4 用户通信流程

一次完整的用户交互链路：

  1. 注册：客户端输入用户名、密码，请求经 Nginx 分发到某台 Lucky 服务器。服务器校验用户名唯一性后写入 MySQL，返回注册成功。
  2. 登录：验证用户名密码，验证通过后将用户状态写入 Redis（key 为 user:{userId}:status，value 为 online），并返回用户信息和好友列表。
  3. 主界面：客户端展示好友列表（从 MySQL 拉取好友关系 + 从 Redis 拉取在线状态）。
  4. 发送消息：选择好友进入聊天界面，输入消息。消息通过 Lucky TCP 连接发送到服务器，服务器根据目标用户所在的服务器进行直接转发或 Redis 发布。
  5. 接收消息：目标用户通过长连接实时收到消息推送，聊天窗口自动更新。
  6. 登出：清理会话，将用户状态更新为 offline。

七、关键设计原则总结

分层分离：网络库只负责 IO 事件分发，业务逻辑通过回调注入。网络层不知道上层在跑 HTTP 还是自定义协议，上层不关心底层用了 epoll 还是 select。接口只有两个：setConnectionCallback 和 setMessageCallback。

无锁优先：One Loop per Thread 的核心价值不在于”用了多线程”，而在于”每个连接只在一个线程里处理”。无锁比加锁快不止一个数量级。跨线程通信通过 EventLoop 的 pending functors 队列，采用短锁 + 交换式取任务的设计，锁持有时间仅几次指针交换。

异步非阻塞：从 epoll 事件驱动、非阻塞 socket IO，到双缓冲日志的异步刷盘，整个 Lucky 的 IO 路径上没有任何阻塞等待。每个组件要么在等待事件（epoll_wait），要么在处理事件（回调），要么在 pending（functors 队列）。

零拷贝与缓存友好：Buffer 的 prependable 空间设计减少了数据搬移次数。readFd 使用 readv 配合栈缓冲区避免了频繁的 vector resize。Filed 通过 data.ptr 直接索引对象，避免了 fd 到对象的映射查找。

自动资源管理：Socket 和 Filed 都遵循 RAII 原则。构造时注册，析构时自动注销。Connection 使用 shared_from_this 管理生命周期，确保在回调执行期间对象不会被意外释放。
